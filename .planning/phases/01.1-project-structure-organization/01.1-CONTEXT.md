# Phase 1.1: Project Structure & Organization - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish project-wide structures that will guide all downstream implementation phases. This includes:
- Directory/file organization for frontend (TanStack Start), backend (ElysiaJS), and shared code
- Database schema foundation with core patterns (timestamps, primary keys, soft deletes)
- API contract organization, error handling patterns, and response envelopes
- Naming conventions for files, folders, database entities, and code
- Integration points and architectural patterns for Phase 2 onward

This phase establishes the structural groundwork for the full application. Monitor-specific tables, API endpoints, and incident handling are deferred to their respective phases (Phase 2 onward).

</domain>

<decisions>
## Implementation Decisions

### Directory Organization
- **D-01:** Agent's discretion on directory layout — plan according to TanStack Start conventions and ElysiaJS patterns for clear separation of concerns

### Database Schema Foundation
- **D-02:** Establish core patterns in Phase 1.1 (timestamps, soft deletes, primary key strategy); full entity list and relationships deferred to Phase 2
- **D-03:** Pattern decisions are the focus: how to handle created_at/updated_at columns, deletion strategies, relationship modeling
- **D-04:** Phase 2 will add monitoring-specific entities (monitors, checks); Phase 3 will add incident tracking entities; later phases will extend as needed

### API Route Structure
- **D-05:** Agent's discretion on versioning approach — establish a common response envelope pattern and route structure that supports growth
- **D-06:** Plan should clarify whether to use /api/v1 from the start or flat /api structure with migration flexibility

### Naming Conventions
- **D-07:** Follow framework conventions — Drizzle ORM standards for database naming, TanStack Start conventions for files/folders, TypeScript standards for types
- **D-08:** Agent should clarify the specific conventions (e.g., camelCase vs PascalCase for table names per Drizzle, kebab-case vs camelCase for components per TanStack)

### Error Handling & Response Patterns
- **D-09:** Agent's discretion on error pattern — establish patterns for common errors (authentication, validation, server errors) based on ElysiaJS best practices
- **D-10:** Create a standard response envelope or pattern that all API endpoints will follow; planner will ensure consistency

### the agent's Discretion
- Exact directory structure layout (will follow framework best practices)
- Specific database schema optimizations (agent advises based on patterns)
- API versioning decision (agent recommends based on scalability needs)
- Exact naming convention specifics (agent applies framework conventions)
- Error response format details (agent applies ElysiaJS standards)

</decisions>

<code_context>
## Existing Code Insights

### Codebase Status
The Livenex codebase is not yet initialized — Phase 1.1 establishes the foundation for Phase 2+ implementation.

### Technology Setup
- **Frontend:** TanStack Start with React 19 (routing, SSR, private/public page separation)
- **Backend:** ElysiaJS API server (lightweight, optimized for Bun/Node)
- **Database:** PostgreSQL 18 + Drizzle ORM (type-safe migrations)
- **Styling:** Tailwind CSS + shadcn/ui component library
- **Auth:** better-auth (handles sessions, optional OIDC)

### Design Considerations for Phase 1.1
- TanStack Start handles page-level routing and private/public separation (Phase 1 auth → Phase 5 public pages)
- ElysiaJS will host all API endpoints; Phase 1 auth endpoints + Phase 2+ domain endpoints
- Drizzle migrations ensure schema evolution is reviewable and version-controllable
- shadcn/ui provides component foundation; naming/structure decisions affect component library usage

</code_context>

<specifics>
## Specific Ideas

- **Scalability for 20-50 monitors:** Structure should not require significant refactoring when adding more monitor types, checks, or incident tracking
- **Single-user simplicity:** Structure should reflect single-user focus (no multi-tenant overhead, no environment complexity)
- **Automation-first API:** API structure should prioritize API key-based access as primary automation interface

</specifics>
