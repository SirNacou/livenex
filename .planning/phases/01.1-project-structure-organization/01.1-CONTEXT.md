# Phase 1.1: Project Structure & Organization - Context

**Gathered:** 2026-03-27
**Status:** Ready for planning

<domain>
## Phase Boundary

Establish project-wide structures that will guide all downstream implementation phases. This includes:
- Directory/file organization for frontend (TanStack Start), backend (ElysiaJS), and shared code
- Database schema foundation with core patterns (timestamps, primary keys, soft deletes)
- API contract organization, error handling patterns, and response envelopes
- Naming conventions for files, folders, database entities, and code
- Integration points and architectural patterns for Phase 2 onward

This phase establishes the structural groundwork for the full application. Monitor-specific tables, API endpoints, and incident handling are deferred to their respective phases (Phase 2 onward).

</domain>

### API Route Structure
- **D-05:** Agent's discretion on versioning approach — establish a common response envelope pattern and route structure that supports growth
- **D-06:** Plan should clarify whether to use /api/v1 from the start or flat /api structure with migration flexibility

### Naming Conventions
- **D-07:** Follow framework conventions — Drizzle ORM standards for database naming, TanStack Start conventions for files/folders, TypeScript standards for types
- **D-08:** Agent should clarify the specific conventions (e.g., camelCase vs PascalCase for table names per Drizzle, kebab-case vs camelCase for components per TanStack)

### Error Handling & Response Patterns
- **D-09:** Agent's discretion on error pattern — establish patterns for common errors (authentication, validation, server errors) based on ElysiaJS best practices
- **D-10:** Create a standard response envelope or pattern that all API endpoints will follow; planner will ensure consistency

### Deployment & Container Strategy
- **D-11:** Frontend and backend deployed as separate Docker containers orchestrated with docker-compose for local development and Kubernetes for production
- **D-12:** TanStack Start frontend runs in its own Node.js/Bun container; ElysiaJS backend runs in separate Bun container
- **D-13:** Monorepo structure with separate build outputs: frontend builds to dist/ for SSR server, backend to dist/api/ for Bun runtime
- **D-14:** Caddy reverse proxy routes requests to both containers: `/` and page routes to frontend, `/api/*` routes to backend
- **D-15:** Frontend accesses backend via same domain (e.g., GET `/api/monitors` resolved by Caddy). No CORS configuration needed (same-origin requests from browser perspective)
- **D-16:** Dockerfile and docker-compose.yml created in Phase 1.1; Caddyfile for local dev; k8s Ingress controller (can use Caddy or other controller) for production
- **D-17:** Environment configuration via .env files for local dev, ConfigMap/Secrets for k8s (database URL, API endpoint, secrets)
- **D-18:** Bun used as runtime for ElysiaJS backend (primary); frontend may use Node or Bun depending on TanStack Start SSR requirements
- **D-19:** Port mapping: frontend 3000 (internal), backend 3001 (internal), Caddy 80/443 (external) — only Caddy exposed to users

<code_context>
## Existing Code Insights

### Codebase Status
The Livenex codebase is not yet initialized — Phase 1.1 establishes the foundation for Phase 2+ implementation.

### Technology Setup
- **Frontend:** TanStack Start with React 19 running on Node.js/Bun (routing, SSR, private/public page separation)
- **Backend:** ElysiaJS API server optimized for Bun runtime (lightweight, fast)
- **Database:** PostgreSQL 18 + Drizzle ORM (type-safe migrations)
- **Styling:** Tailwind CSS + shadcn/ui component library
- **Auth:** better-auth (handles sessions, optional OIDC)
- **Runtime:** Bun for ElysiaJS (primary); Node/Bun for TanStack Start depending on requirements

### Deployment Infrastructure
- **Local Development:** docker-compose.yml with 4 services: Caddy reverse proxy (:80/:443), TanStack Start frontend (:3000 internal), ElysiaJS backend (:3001 internal), PostgreSQL
- **Production:** Kubernetes with frontend Deployment, backend Deployment, PostgreSQL StatefulSet, Ingress controller as reverse proxy
- **Container Strategy:** Separate Dockerfiles for frontend and backend; monorepo structure with independent build outputs
- **Reverse Proxy (Caddy):** Routes `/` and page routes to frontend container; `/api/*` to backend container; automatic HTTPS, TLS termination, X-Forwarded-* headers
- **Same-Domain Architecture:** Frontend calls `/api/monitors` (relative URL); Caddy resolves to backend container internally. No CORS needed (same-origin from browser perspective)
- **Internal Communication:** Backend URL stored in env (e.g., BACKEND_INTERNAL_URL for any inter-service calls); frontend always uses relative paths
- **Environment:** .env files for dev, k8s ConfigMap/Secrets for prod (DATABASE_URL, BACKEND_INTERNAL_URL for inter-container communication)

### Design Considerations for Phase 1.1
- TanStack Start handles page-level routing and private/public separation (Phase 1 auth → Phase 5 public pages)
- ElysiaJS will host all API endpoints; Phase 1 auth endpoints + Phase 2+ domain endpoints
- Bun optimization for ElysiaJS enables fast API performance on self-hosted infrastructure
- Drizzle migrations ensure schema evolution is reviewable and version-controllable
- Caddy as reverse proxy: automatic HTTPS, simple Caddyfile configuration, easy to maintain and extend
- Docker/K8s structure enables independent frontend/backend scaling while maintaining same-domain UX
- shadcn/ui provides component foundation; naming/structure decisions affect component library usage

</code_context>

<specifics>
## Specific Ideas

- **Scalability for 20-50 monitors:** Structure should not require significant refactoring when adding more monitor types, checks, or incident tracking
- **Single-user simplicity:** Structure should reflect single-user focus (no multi-tenant overhead, no environment complexity)
- **Automation-first API:** API structure should prioritize API key-based access as primary automation interface
- **Self-hosted deployment:** Docker Compose for local development, Kubernetes for production deployment on self-hosted infrastructure
- **Bun runtime optimization:** ElysiaJS should leverage Bun's performance advantages; structure supports fast iteration and deployment
- **Caddy reverse proxy:** Use Caddy for automatic HTTPS, simple configuration, and easy maintenance in both local and production environments

</specifics>
