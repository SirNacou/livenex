## Objective

Establish the architectural foundation for Livenex by creating a standard project structure, database schema patterns, API route organization, naming conventions, and error handling that downstream phases will build upon.

This phase runs parallel to Phase 1 planning (not blocking it) and establishes infrastructure decisions that prevent refactoring during Phase 2-5 implementation.

## Plans Created

### 01.1-01-PLAN.md — Project Structure & Foundation

**Wave:** 1 (parallel)  
**Tasks:** 4 autonomous tasks  
**Estimated Duration:** 3-4 hours agent execution  

**Key deliverables:**

1. **Task 1: Project Structure & package.json**
   - Directory layout following TanStack Start + ElysiaJS conventions
   - Separated frontend (client), backend (api), shared (lib, types, db)
   - package.json with all core and dev dependencies from STACK research
   - tsconfig.json configured for strict TypeScript
   - app.config.ts for TanStack Start entry points

2. **Task 2: Database Schema Foundation**
   - Core tables: users, sessions, api_keys, audit_logs
   - Drizzle ORM schema definitions with TypeScript types
   - UUID primary keys with uuid_generate_v4() defaults
   - Timestamps pattern: created_at, updated_at on all tables
   - Soft delete pattern: deleted_at column for future soft-delete behavior
   - Initial migration generated and ready for pushdb

3. **Task 3: API Route Structure & Response Envelopes**
   - ElysiaJS app.ts with global error handler
   - ApiResponse<T> success envelope (success, data, timestamp)
   - ApiError error envelope (success, error with code/message/details, timestamp)
   - Error hierarchy: ValidationError, AuthenticationError, NotFoundError, ConflictError, InternalError
   - Health check endpoint (/health) verifying API structure
   - Response wrapper helper for consistent formatting

4. **Task 4: Documentation & Patterns**
   - CONVENTIONS.md: Database, code, and API naming standards
   - ARCHITECTURE.md: Layer boundaries, error handling, query patterns
   - src/db/SCHEMA-PATTERNS.md: Templates for timestamps, soft deletes, relationships, enums
   - src/api/ROUTE-PATTERNS.md: Templates for CRUD, list with pagination, action endpoints

## Architecture Decisions

**Per Phase 1.1 CONTEXT.md Decisions:**

- **D-01 (Directory Organization):** Agent applied TanStack Start + ElysiaJS best practices with clear separation of concerns
- **D-02, D-03 (Database Patterns):** UUID primary keys, explicit timestamps, soft-delete column established; monitoring-specific entities deferred to Phase 2
- **D-05, D-06 (API Route Structure):** Flat /api structure (no /api/v1 yet) with unified response envelope, future-proof for versioning
- **D-07, D-08 (Naming Conventions):** snake_case for SQL (Drizzle), camelCase for code, kebab-case for files; documented in CONVENTIONS.md
- **D-09, D-10 (Error Handling):** AppError hierarchy with specific codes (VALIDATION_ERROR, UNAUTHORIZED, NOT_FOUND, etc.); global middleware ensures all responses follow envelope
