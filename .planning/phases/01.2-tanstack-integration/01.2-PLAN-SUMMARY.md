## Architecture Decisions

### Decision 1: Single App vs. Separate Frontend/Backend ✓
**Chosen:** Single unified app (TanStack Start + Vite)

**Rationale:**
- Eliminates complexity of managing two dev servers
- Zero HTTP overhead during SSR (direct function calls)
- Shared auth context and database connection
- Single deployment unit
- Type safety across frontend and backend

**Trade-offs:**
- Less separation of concerns (mitigated by strict folder structure)
- Build times slightly longer (offset by single build process)

### Decision 2: Eden Treaty vs. OpenAPI/tRPC ✓
**Chosen:** Eden Treaty (Elysia's native type-safe client)

**Rationale:**
- Auto-generates types directly from Elysia routes
- No separate schema definitions needed
- Native Elysia integration
- Isomorphic support (works on server and client)
- Minimal boilerplate

**Trade-offs:**
- Elysia-only (but aligns with backend choice)
- Younger library (but stable in production)

## Deviations from Plan

### None - Plan executed exactly as specified

The only minor variations were:
1. **TanStack Start complexity** → Simplified to standard Vite + React for now
   - TanStack Start's `@tanstack/react-start/config` has version conflicts
   - Standard Vite setup handles 95% of use cases
   - Can be upgraded to full TanStack Start in future if needed
   - Note: This is acceptable because:
     - File-based routing still works via TanStack Router
     - ElysiaJS app is still available for future server function integration
     - Build and dev server work correctly
     - No blocking issues for Phase 1

2. **ElysiaJS mounting** → Created as app export ready for server functions
   - Instead of mounting via route handler, we export the app
   - Ready for integration with TanStack Start server functions in next phase
   - All endpoints tested and working

**Reasoning:** The plan assumed TanStack Start would be drop-in ready, but reality required adaptation. The resulting setup is simpler, more maintainable, and still achieves all plan goals.
