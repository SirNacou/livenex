<objective>
Consolidate separate frontend/backend packages into a single unified TanStack Start + ElysiaJS full-stack application.

Purpose: Eliminate the complexity of managing two separate dev servers, deployments, and HTTP overhead during SSR while maintaining clean architectural boundaries between client and server code.

Output: Single, deployable Node.js application with:
- One dev server handling React routes + ElysiaJS API routes
- Shared database connection and auth context
- Type-safe frontend-to-backend communication via Eden Treaty
- Zero HTTP overhead on server during SSR
</objective>

Create type-safe frontend-to-backend bridge:

1. **Create src/lib/api.ts (Eden Treaty setup):**
   ```typescript
   import { treaty } from '@elysiajs/eden'
   import { createIsomorphicFn } from '@tanstack/react-start'
   import { app } from '~/server/api'
   
   export const getTreaty = createIsomorphicFn()
     // Server: direct function call (no HTTP)
     .server(() => treaty(app).api)
     // Client: HTTP call to /api (with JSON serialization)
     .client(() => treaty<typeof app>('http://localhost:3000').api)
   ```

2. **Usage in loaders (zero HTTP during SSR):**
   ```typescript
   // src/routes/dashboard/index.tsx
   export const Route = createFileRoute('/dashboard')({
     beforeLoad: async (context) => {
       // During SSR: getTreaty() → direct function call
       // During client nav: getTreaty() → HTTP request
       const result = await getTreaty().auth.session.get()
       
       if (!result.data) {
        return { redirect: '/auth/login' }
       }
       return { auth: result.data }
     }
   })
   ```

3. **Usage in components (client-side only):**
   ```typescript
   // src/components/ApiKeysList.tsx
   import { useQuery } from '@tanstack/react-query'
   import { getTreaty } from '~/lib/api'
   
   export function ApiKeysList() {
     const { data, isLoading } = useQuery({
       queryKey: ['api-keys'],
       queryFn: () => getTreaty().api.keys.list()
     })
     // ...
   }
   ```

4. **Type inference check:**
   - Hover over `getTreaty().auth.session.get()` in IDE
   - Type should be inferred as `Promise<{ data: Session | null, status: 200 }>`
   - No `any` types should appear

5. **No manual API client code needed:**
   - Eden Treaty auto-generates types from Elysia routes
   - No separate OpenAPI or zod schema exports needed
   - Type safety is automatic

**Why:** Eliminates HTTP overhead during SSR (direct calls) while maintaining type safety. Client-side calls still use HTTP (standard browser limitation) but with full type inference.
